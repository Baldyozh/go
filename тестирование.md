# 5 ТЕСТИРОВАНИЕ И ОПТИМИЗАЦИЯ

## 5.1 План тестирования

Целью тестирования является проверка корректности и устойчивости программного средства обработки логов. Система получает сообщения из Kafka, шифрует чувствительные поля, сохраняет обработанные записи в ClickHouse и предоставляет HTTP API для поиска, фильтрации, просмотра и экспорта логов. Также в системе реализованы авторизация пользователей, ролевая модель доступа и возможность расшифровки конфиденциальных данных при наличии соответствующих прав.

Для данного программного средства наиболее предпочтительными являются следующие виды тестирования.

### Модульное тестирование

Модульное тестирование используется для проверки отдельных компонентов без запуска всей инфраструктуры. Этот вид тестирования является основным для проверки бизнес-логики, так как проект построен по принципам чистой архитектуры: логика обработки, шифрования, фильтрации и управления логами выделена в отдельные пакеты.

В рамках модульного тестирования проверяются:

- корректность шифрования и расшифровки данных алгоритмом AES-GCM;
- обработка ошибочных ключей и поврежденных зашифрованных данных;
- обработка JSON-полей для шифрования;
- работа с глобальными именами полей, например `passport` и `snils`;
- работа с позиционными путями, например `user.profile.passport`;
- работа с массивами и wildcard-путями, например `items.*.secret`;
- обработка некорректного JSON;
- обработка ошибки при попытке зашифровать или расшифровать нестроковое поле;
- проверка прав доступа при просмотре и расшифровке логов;
- проверка формирования CSV-экспорта;
- проверка работы batch writer и worker pool.

Модульное тестирование выбрано потому, что оно позволяет быстро находить ошибки в отдельных частях программы и не требует постоянного запуска Kafka, Vault, ClickHouse и PostgreSQL.

### Интеграционное тестирование

Интеграционное тестирование необходимо, так как программное средство взаимодействует с несколькими внешними сервисами:

- Kafka используется для получения потока логов;
- Vault используется для шифрования чувствительных данных в режиме `vault`;
- ClickHouse используется для хранения и аналитической выборки логов;
- PostgreSQL используется для хранения пользователей, ролей и аудита расшифровки;
- HTTP API используется для взаимодействия пользователя с системой.

В рамках интеграционного тестирования проверяются:

- запуск инфраструктуры через Docker Compose;
- подключение приложения к Kafka;
- чтение сообщений из топика `logs`;
- запись обработанных логов в ClickHouse;
- подключение к PostgreSQL;
- авторизация пользователя и выдача JWT-токена;
- доступ к защищенным API-методам;
- запрет доступа к данным без токена;
- расшифровка чувствительных данных при наличии права `logs:decrypt`.

Интеграционное тестирование является важным, так как корректная работа отдельных модулей не гарантирует корректный обмен данными между сервисами.

### Функциональное тестирование API

Функциональное тестирование проводится для проверки пользовательских сценариев. Основное взаимодействие с программным средством выполняется через HTTP API, поэтому необходимо проверить все основные действия пользователя.

Проверяются следующие сценарии:

- вход пользователя в систему;
- получение информации о текущем пользователе;
- поиск логов по идентификатору запроса;
- фильтрация логов по интеграции, времени, HTTP-методу, endpoint, коду ответа и признаку успешности;
- просмотр детальной информации о конкретном логе;
- запрос расшифровки чувствительных данных;
- экспорт логов в CSV;
- создание нового пользователя администратором.

### Тестирование безопасности

Тестирование безопасности является обязательным, так как система обрабатывает конфиденциальные данные. К таким данным относятся паспортные данные, СНИЛС и другие чувствительные поля, указанные в конфигурации.

Проверяются:

- запрет доступа к защищенным маршрутам без JWT-токена;
- запрет чтения логов без права `logs:read`;
- запрет поиска логов без права `logs:search`;
- запрет фильтрации логов без права `logs:filter`;
- запрет расшифровки без права `logs:decrypt`;
- доступ к созданию пользователей только при наличии права `users:manage`;
- отсутствие хэша пароля в HTTP-ответах;
- фиксация факта и причины расшифровки данных.

### Нагрузочное тестирование

Нагрузочное тестирование используется для оценки производительности программного средства при обработке большого количества логов. Оно особенно важно для данной системы, так как обработка логов выполняется в потоковом режиме.

В рамках нагрузочного тестирования проверяются:

- скорость обработки сообщений;
- влияние количества worker-горутин на пропускную способность;
- влияние размера буферизированного канала на блокировки между этапами обработки;
- влияние размера батча на скорость записи в ClickHouse;
- различие между шифрованием через Vault и локальным шифрованием;
- поведение системы при разных размерах тела лога;
- поведение системы при кратковременных пиках нагрузки.

### Виды тестирования, которые не проводятся

Тестирование пользовательского интерфейса не проводится, так как в текущей реализации программное средство не содержит полноценного графического интерфейса. Основное взаимодействие выполняется через HTTP API.

Кроссбраузерное тестирование не проводится, так как система не зависит от браузера пользователя.

Тестирование на мобильных устройствах не проводится, так как программное средство является серверным приложением и не предназначено для установки на мобильные операционные системы.

A/B-тестирование не проводится, так как система не содержит альтернативных вариантов пользовательского интерфейса или бизнес-логики, которые необходимо сравнивать по поведению пользователей.

Результат выполнения: тесты завершились успешно.

В ходе модульного тестирования были проверены следующие компоненты.

1. **Шифрование AES-GCM**

Проверялась корректность шифрования и расшифровки строковых данных. Для одной и той же исходной строки результат шифрования должен отличаться при повторных запусках, так как при AES-GCM используется случайный nonce.

Также проверялись негативные сценарии:

- создание шифра с ключом неверной длины;
- попытка расшифровки слишком короткого ciphertext;
- попытка расшифровки поврежденного ciphertext;
- попытка расшифровки строки, не являющейся корректным base64.

Результат: компонент шифрования работает корректно и обрабатывает ошибочные входные данные.

2. **Обработка JSON-полей для шифрования и расшифровки**

Проверялась работа с разными способами указания чувствительных полей:

- глобальное имя поля, например `passport`;
- позиционный путь, например `user.passport`;
- смешанная конфигурация глобальных и позиционных полей;
- wildcard для массивов, например `items.*.secret`;
- индекс массива, например `items.1.tokens.0`.

Также проверялись ошибки:

- некорректный JSON;
- отсутствие поля по указанному пути;
- попытка обработать нестроковое значение.

Результат: механизм выбора чувствительных полей работает корректно, ошибочные данные не игнорируются и возвращают ошибку.

3. **Парсинг фильтров логов**

Проверялась функция разбора query-параметров HTTP-запроса. Были проверены параметры:

- `request_id`;
- `integration_id`;
- `start_time`;
- `end_time`;
- `status_code`;
- `http_method`;
- `endpoint`;
- `user_id`;
- `success`.

Также проверялась обработка некорректных значений времени, кода статуса и идентификатора пользователя.

Результат: корректные параметры преобразуются в структуру фильтра, некорректные параметры возвращают ошибку.

4. **Проверка прав доступа в сервисе управления логами**

Проверялись сценарии:

- запрет получения лога без права `logs:read`;
- запрет расшифровки без права `logs:decrypt`;
- успешная расшифровка при наличии нужного права;
- запись причины расшифровки в аудит;
- запрет фильтрации без права `logs:filter`;
- экспорт логов в CSV при наличии права фильтрации.

Результат: ролевая модель доступа работает корректно.

5. **Проверка worker pool и batch writer**

Проверялась логика обработки логов в usecase-слое:

- worker шифрует тело запроса и тело ответа;
- запись пропускается, если на этапе шифрования произошла ошибка;
- batch writer выполняет запись при достижении заданного размера батча;
- batch writer записывает оставшиеся данные при завершении работы.

Результат: логика параллельной обработки и пакетной записи работает корректно.

### Оптимизация

По результатам тестирования была проведена оптимизация программного средства. Основная цель оптимизации заключалась в увеличении пропускной способности обработки логов и снижении задержек между чтением сообщений из Kafka, шифрованием и записью данных в ClickHouse.

Оптимизация была выполнена за счет изменения конфигурации приложения.

#### Добавление конфигурации с локальным шифрованием

В исходной конфигурации использовалось шифрование через Vault:

```yaml
crypto:
  encryption_type: "vault"
```

Такой вариант обеспечивает централизованное управление ключами, но при высокой нагрузке каждое обращение к Vault создает дополнительную сетевую задержку. Поэтому была добавлена отдельная конфигурация `config/config-local.yaml`, в которой используется локальное шифрование:

```yaml
crypto:
  encryption_type: "local"
  local_encryption:
    key_rotation_interval: "24h"
    algorithm: "aes-gcm"
```

В локальном режиме шифрование выполняется внутри приложения с использованием алгоритма AES-GCM. Это уменьшает количество сетевых обращений и снижает время обработки одного сообщения.

Преимущества локального шифрования:

- уменьшается задержка обработки лога;
- снижается зависимость производительности от доступности Vault;
- уменьшается нагрузка на сеть;
- повышается стабильность обработки при больших объемах данных.

При этом чувствительные поля продолжают шифроваться, а список таких полей остается настраиваемым:

```yaml
fields_to_encrypt:
  - "passport"
  - "snils"
```

#### Увеличение количества горутин в worker pool

В системе используется worker pool, который параллельно обрабатывает входящие логи. Каждый worker получает запись, шифрует чувствительные поля в теле запроса и ответа, после чего передает результат на запись в ClickHouse.

В базовой конфигурации количество worker-горутин составляло:

```yaml
worker_pool:
  worker_count: 4
```

В оптимизированной конфигурации количество worker-горутин было увеличено:

```yaml
worker_pool:
  worker_count: 6
```

Увеличение количества worker-горутин позволяет параллельно обрабатывать больше сообщений. Это особенно полезно при высокой нагрузке, когда в Kafka поступает большой поток логов.

Результат оптимизации:

- увеличивается пропускная способность обработки;
- уменьшается время ожидания логов в очереди;
- эффективнее используются ресурсы процессора;
- снижается риск накопления необработанных сообщений.

#### Увеличение размера буферизированного канала для логов

Между этапами обработки используются буферизированные каналы. Они позволяют временно накапливать сообщения и уменьшают блокировки между чтением логов, шифрованием и записью.

В базовой конфигурации размер буфера составлял:

```yaml
worker_pool:
  channel_buffer_size: 100
```

В оптимизированной конфигурации размер буфера был увеличен:

```yaml
worker_pool:
  channel_buffer_size: 300
```

Увеличение буфера позволяет системе лучше переносить кратковременные пики нагрузки. Если сообщения поступают из Kafka неравномерно, увеличенный канал дает worker pool возможность продолжать обработку без постоянной блокировки чтения.

Результат оптимизации:

- уменьшается количество блокировок между этапами pipeline;
- сглаживаются кратковременные всплески нагрузки;
- повышается устойчивость обработки;
- worker-горутины получают более равномерный поток задач.

#### Изменение размера батча для записи в ClickHouse

ClickHouse эффективнее работает с пакетной вставкой данных, чем с записью каждой строки отдельно. Поэтому в приложении используется batch writer, который накапливает несколько логов и записывает их одной операцией.

В оптимизированной конфигурации размер батча установлен следующим образом:

```yaml
batch:
  size: 5000
  flush_interval: "5s"
```

Размер батча `5000` позволяет уменьшить количество операций записи в ClickHouse. При этом параметр `flush_interval: "5s"` ограничивает максимальное время ожидания записи: если батч не успел заполниться полностью, накопленные данные все равно будут записаны по таймеру.

Результат оптимизации:

- уменьшается количество сетевых обращений к ClickHouse;
- снижаются накладные расходы на вставку данных;
- увеличивается скорость записи логов;
- ClickHouse эффективнее обрабатывает поток данных;
- сохраняется баланс между пропускной способностью и задержкой записи.
