## Слайд 2: Актуальность

В современных условиях цифровой трансформации компании активно интегрируются с внешними API-партнёрами. Каждая такая интеграция генерирует огромные объёмы логов, которые содержат критически важную информацию для отладки, мониторинга и анализа.

Однако здесь возникает ряд серьёзных проблем:

**Во-первых**, рост количества интеграций приводит к экспоненциальному увеличению объёмов логов. Традиционные системы логирования не справляются с такой нагрузкой.

**Во-вторых**, логи часто содержат чувствительные данные: персональную информацию, финансовые данные, медицинские записи. Хранение таких данных в открытом виде создаёт серьёзные риски безопасности.

**В-третьих**, бизнес требует высокой скорости обработки логов для оперативного реагирования на инциденты.

**И в-четвёртых**, Федеральный закон №152-ФЗ "О персональных данных" обязывает компании шифровать персональные данные при хранении.

Моё решение - это распределённый процессор логов с автоматическим шифрованием чувствительных данных, который решает все эти проблемы.

## Слайд 4: Аналоги и конкурентные преимущества

На рынке существуют различные решения для логирования:

**GrayLog** - это open-source решение, но оно требует дополнительной настройки для обеспечения шифрования данных, что усложняет внедрение.

**Splunk** - мощная коммерческая система, но её лицензии стоят очень дорого, что делает её недоступной для многих компаний.

**ELK Stack** - популярный набор инструментов, но его настройка безопасности требует значительных усилий и экспертизы.

**Моё решение имеет ключевые преимущества**:

**Встроенное шифрование** - автоматическое шифрование чувствительных полей "из коробки", без дополнительной настройки.

**Высокая производительность** - комбинация ClickHouse и batch processing обеспечивает скорость обработки около 10,000 логов в секунду.

**Масштабируемость** - архитектура с worker pool и Kafka позволяет горизонтально масштабировать систему.

**Соответствие ФЗ-152** - полное соответствие требованиям российского законодательства о защите персональных данных.

**Открытый исходный код** - полный контроль над системой и возможность кастомизации под нужды компании.

## Слайд 5: Архитектура системы

Давайте рассмотрим архитектуру разработанной системы. Она состоит из шести основных компонентов:

1. **Kafka Consumer** - отвечает за приём логов из очереди сообщений Kafka, что обеспечивает отказоустойчивость и буферизацию при пиковых нагрузках.

2. **Log Processor** - основной компонент системы, который использует worker pool для параллельной обработки логов.

3. **Vault Integration** - интеграция с HashiCorp Vault для безопасного шифрования и дешифрования данных.

4. **ClickHouse** - колоночная СУБД для хранения зашифрованных логов с возможностью быстрого поиска и аналитики.

5. **REST API** - предоставляет безопасный доступ к логам с возможностью дешифрования по запросу.

6. **PostgreSQL** - хранит информацию о пользователях и их правах доступа.

**Поток данных** выглядит следующим образом: логи поступают в Kafka, затем обрабатываются процессором, шифруются через Vault, сохраняются в ClickHouse, и через REST API становятся доступны авторизованным пользователям с автоматическим дешифрованием.

## Слайд 7: Процесс шифрования и дешифрования

Теперь о самом важном - процессе шифрования данных.

**При записи логов** система выполняет следующие шаги:
1. Получает JSON-лог из Kafka
2. Парсит чувствительные поля согласно конфигурации
3. Отправляет данные в Vault Transit Engine для шифрования
4. Получает зашифрованные данные с метаданными о версии ключа
5. Записывает в ClickHouse, сохраняя исходную структуру JSON

**При чтении логов** через REST API:
1. Пользователь отправляет запрос с JWT-токеном
2. Система проверяет права доступа
3. Извлекает зашифрованные поля из ClickHouse
4. Отправляет их в Vault для дешифрования
5. Возвращает расшифрованные данные авторизованному пользователю

Система поддерживает различные типы полей:
- Глобальные поля вроде "email" или "phone"
- Вложенные поля типа "user.personal_data.ssn"
- Массивы данных с индексацией

Это обеспечивает гибкость в настройке шифрования под различные форматы логов.

## Слайд 9: Демонстрация системы

Система предоставляет REST API с следующими эндпоинтами:

- `POST /api/auth/login` для аутентификации пользователей
- `GET /api/logs` для поиска логов с гибкой фильтрацией
- `GET /api/logs/{id}` для получения конкретного лога с автоматическим дешифрованием
- `GET /api/logs/stats` для получения статистики по логам

Например, можно найти все логи конкретной интеграции за определённый период с определённым статусом HTTP.

В ответе система автоматически дешифрует чувствительные поля, предоставляя пользователю полную информацию в удобном формате.
